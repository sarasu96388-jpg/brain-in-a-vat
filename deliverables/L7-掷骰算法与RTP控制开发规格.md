## 2. 掷骰子核心算法

### 2.1 算法总览：先定结果，再反推骰子

```
┌─────────────────────────────────────────────────────────┐
│  Step 1: 确定目标落点格                                    │
│  → 根据当前格子位置 + 格子权重 + 奖池状态 + 保底机制         │
│  → 从前方2-12格范围内选择目标格                            │
├─────────────────────────────────────────────────────────┤
│  Step 2: 反推骰子点数和                                    │
│  → 目标格编号 - 当前格编号 = 需要的点数和(2~12)            │
│  → 如果差值不在[2,12]范围内，回退到Step 1重新选格           │
├─────────────────────────────────────────────────────────┤
│  Step 3: 随机化具体骰子值                                  │
│  → 根据点数和，随机分配两个骰子的具体值                      │
│  → 例：点数和=7 → 随机[1,6]/[2,5]/[3,4]等                 │
├─────────────────────────────────────────────────────────┤
│  Step 4: 执行移动 + 触发格子事件                           │
│  → 播放骰子动画 → 移动到目标格 → 结算奖励                   │
└─────────────────────────────────────────────────────────┘
```

### 2.2 Step 1 详细逻辑：目标格选择

#### 2.2.1 候选范围

- 从当前格前方第2格到第12格（双骰子最小2、最大12）
- 如果前方不足12格到达终点，则候选范围为当前格前方第2格到终点格

#### 2.2.2 权重计算

每个候选格的基础权重来自配置表，经过以下修正：

```
最终权重 = 基础权重 × 奖池修正系数 × 保底修正系数

奖池修正系数：
  - 如果该格类型在当前生效奖池(+)的加分事件中 → 权重提升
  - 如果该格类型在当前生效抽水池(-)的抽水事件中 → 权重降低
  - 具体修正值见第4章

保底修正系数：
  - 如果该格类型触发保底机制 → 权重临时大幅提升
  - 具体规则见第5章
```

#### 2.2.3 选择算法

```
1. 计算所有候选格的最终权重
2. 归一化为概率分布（各格概率 = 该格权重 / 总权重）
3. 按概率随机选择一个目标格
```

### 2.3 Step 2 详细逻辑：反推点数和

```
distance = targetTileId - currentTileId

if distance < 2 or distance > 12:
    // 不可达，回退重新选格
    return Step 1(retry)

diceSum = distance
```

**防死循环保护**：如果连续3次Step 1选出的格都不可达，强制选择距离当前格最近的合法格（优先选distance=6或7，因为概率最高）。

### 2.4 Step 3 详细逻辑：骰子值随机化

```
// 根据diceSum生成两个骰子的具体值
function generateDiceValues(diceSum):
    validPairs = []
    for d1 in 1..6:
        d2 = diceSum - d1
        if d2 >= 1 and d2 <= 6:
            validPairs.append([d1, d2])
    
    // 均匀随机选择一对
    return randomChoice(validPairs)
```

| 点数和 | 可能的骰子组合 | 组合数 |
|--------|--------------|--------|
| 2 | [1,1] | 1 |
| 3 | [1,2],[2,1] | 2 |
| 4 | [1,3],[2,2],[3,1] | 3 |
| 5 | [1,4],[2,3],[3,2],[4,1] | 4 |
| 6 | [1,5],[2,4],[3,3],[4,2],[5,1] | 5 |
| 7 | [1,6],[2,5],[3,4],[4,3],[5,2],[6,1] | 6 |
| 8 | [2,6],[3,5],[4,4],[5,3],[6,2] | 5 |
| 9 | [3,6],[4,5],[5,4],[6,3] | 4 |
| 10 | [4,6],[5,5],[6,4] | 3 |
| 11 | [5,6],[6,5] | 2 |
| 12 | [6,6] | 1 |

### 2.5 双骰子(Double)判定

```
isDouble = (dice1 == dice2)

if isDouble:
    // 触发Double奖励（额外掷骰机会或其他奖励，按配置）
    // 连续Double的处理规则按具体活动配置
```

## 3. RTP控制系统

### 3.1 RTP定义

```
RTP = total_outing（总产出价值）/ total_input（总投入价值）

total_input = 玩家消耗的骰子总价值（骰子数量 × 单骰子基准价值）
total_outing = 玩家从棋盘获得的所有奖励总价值
```

### 3.2 RTP控制的两层架构

L7的RTP拆解为两个层面，确保既有底线（不亏本），又有上限（不通胀）。

#### 3.2.1 基础RTP（底线守卫者）

| 属性 | 说明 |
|------|------|
| 来源 | 由棋盘60格的原始权重决定 |
| 设计目标 | 长期RTP < 100%（建议设定在 92%-95%） |
| 角色 | 系统的"重力"。当奖池和抽水池均为空时，玩家回归此权重 |
| 价值回收 | 通过自然的金币/骰子损耗实现 |
| 调整时机 | 仅在测试期调整，上线后不应改动 |

**核心原则**：基础权重的设计必须保证没有奖池加持时，玩家会逐渐消耗骰子。这是整个系统的底线。

#### 3.2.2 短期RTP（体感控制器）

| 属性 | 说明 |
|------|------|
| 决定因素 | 奖池(+)和抽水池(-)系统 |
| 目标 | 围绕基础RTP制造波动，塑造幸运/倒霉体验 |
| 控制方式 | 动态调整格子触发概率和事件奖励倍率 |

### 3.3 短期RTP概率提升公式

```
P_boost = P_base × (1 + K × V_pool / (E_val × Bet))

参数说明：
  P_boost  = 提升后的事件触发概率
  P_base   = 事件基础触发概率（来自格子权重配置）
  K        = 调节系数（配置值，控制概率提升的敏感度）
  V_pool   = 当前奖池剩余分值（水位）
  E_val    = 事件平均期望分值
  Bet      = 玩家当前bet倍率对应的下注价值

核心逻辑：
  - 概率提升倍率与奖池水位成正比（奖池越满，提升越大）
  - 与"单次预期消耗（E_val × Bet）"成反比（bet越高，提升越温和）
```

**水位调节效果**：

| 场景 | V_pool | Bet | P_boost效果 |
|------|--------|-----|------------|
| 满池+低bet | 高 | 低 | 大幅提升 |
| 满池+高bet | 高 | 高 | 适度提升 |
| 空池+任意 | 接近0 | 任意 | 回归基础概率 |

**绝对上限保护**：
```
P_boost ≤ min(P_base × 3.0, 0.5)
```

### 3.4 事件期望分值动态计算

```
E_val = Gold × 1.0 + Token × Rate_token + Dice × Rate_dice

参数说明：
  Gold        = 金币奖励期望值
  Token       = 活动代币期望值
  Dice        = 骰子奖励期望值
  Rate_token  = 代币动态价值系数（随活动阶段变化）
  Rate_dice   = 骰子动态价值系数（随玩家存量变化）
```

**动态价值系数的作用**：当Top Bar产出变得极其慷慨时，E_val的分母变大，P_boost自动压低概率，防止瞬间崩盘。

### 3.5 短期RTP上调触发条件

| 触发场景 | 设计目的 | 持续时间 |
|---------|---------|---------|
| 新地图开启 | 新鲜感+正向激励，留住玩家 | 配置值（通常前20-50骰） |
| 充值后 | 充值正反馈，鼓励继续付费 | 配置值 |
| 破产恢复 | 防止流失，给翻本机会 | 配置值 |
| 活动即将完成 | 临门一脚，促进活动completion | 活动剩余10%-20%进度 |
| 倍率从高调低 | 平滑过渡，避免骤降感 | 倍率切换后的过渡期 |

### 3.6 短期RTP下调触发条件

| 触发场景 | 设计目的 | 持续时间 |
|---------|---------|---------|
| 活动即将完成 | 制造难度，刺激付费冲刺 | 活动最后阶段 |
| 资源存量过高 | 防止玩家攒太多资源导致后续付费失效 | 持续监控 |

### 3.7 周期性RTP回调机制

```
每个周期结束时计算：
    实际RTP = 本周期total_outing / 本周期total_input
    目标RTP = 基础RTP基准值
    偏离度 = (实际RTP - 目标RTP) / 目标RTP

如果 偏离度 > +容忍阈值（如+15%）：
    下周期降低奖池触发频率 5%~10%
    
如果 偏离度 < -容忍阈值（如-15%）：
    下周期提高奖池触发频率 5%~10%

调整幅度上限：单次调整不超过±10%，防止剧烈波动
```

### 3.8 反推算法优化：保底步数计数器

由于算法是"先定落点再反推骰子值"，在P_boost极高时（如低bet玩家磨池子），可能会出现连续多次掷骰都落在大奖格的情况。

**机制**：引入保底步数计数器，防止"必中Bug"的廉价感。

```
为奖池关联事件类型维护连续触发计数器 consecutiveHits[eventType]

每次掷骰后：
  如果落点是奖池关联格子 → consecutiveHits[eventType]++
  否则 → consecutiveHits[eventType] = 0

当 consecutiveHits[eventType] >= 3 时：
  下次掷骰Step 1选格时，强制本次落点为"平庸格"
  （普通金币格、低价值特殊关联格等）
  consecutiveHits[eventType] = 0

目的：
  - 避免玩家产生"系统出Bug了必赢"的廉价感
  - 保持中奖的惊喜感
  - 不影响整体RTP控制（只是打散连续中奖）
```

## 4. 奖池(+)和抽水池(-)系统

### 4.1 核心概念

**奖池(+)**：存储"希望额外给玩家的价值"。池子存在期间，对应事件触发概率按P_boost公式提升，玩家高频拿到好奖励，塑造幸运体验。

**抽水池(-)**：存储"希望额外从玩家回收的价值"。仅作为新手期后的难度调节工具，不由玩家赢多了触发，而由资源存量触发。

### 4.2 奖池数据结构

```
PrizePool {
    poolId: string              // 唯一标识
    poolType: "+" | "-"         // 奖池或抽水池
    priority: T0 | T1 | T2      // 优先级
    triggerCondition: string    // 触发条件（充值/新地图/破产等）
    
    // 分值分配（按事件类型分饼）
    allocations: {
        eventType: string       // 事件类型（铁路/空艇/骰子/钱1/钱2/钱3等）
        value: number           // 分配给该事件的分值（额外价值预算）
    }[]
    
    // 状态
    remainingValue: number      // 剩余总分值
    initialValue: number        // 初始总分值（含5%溢出预留）
    initialDiceCount: number    // 生效时的掷骰计数
    maxDiceCount: number        // 最大存活掷骰次数（死亡条件之一）
    isActive: boolean           // 是否当前生效
    
    // 体验控制参数
    minValue: number            // Min阈值（事件产出价值下限）
    maxValue: number            // Max阈值（事件产出价值上限）
    alpha: number               // 低产出加速系数（0.3~0.5）
    beta: number                // 高产出缓释系数（0.3~0.5）
    K: number                   // P_boost公式调节系数
    
    // 坏账预留
    overflowReserve: number     // 溢出预留（默认initialValue的5%）
}
```

### 4.3 奖池入队与生效规则

#### 4.3.1 优先级队列

```
优先级顺序：T0 > T1 > T2
同优先级内：奖池(+) 优先于 抽水池(-)
同级多池：按入队先后顺序
```

#### 4.3.2 生效规则

```
1. 每次掷骰前，检查当前优先级队列
2. 按优先级从高到低，同优先级先奖池后抽水池
3. 同一时间只有一个奖池(+)和一个抽水池(-)处于"当前生效"状态
4. 未生效的奖池封存，不进入消亡倒计时
5. 当前生效池死亡后，从队列中取出下一个优先级的池激活
```

#### 4.3.3 多池并行处理

```
如果同一时间存在同优先级的多个奖池：
  ① 概率提升直接叠加
  ② 奖金浮动倍率取相对较高值
  ③ 触发时从奖池中根据先后顺序分配抽取额度

示例：
  T0级别同时存在奖池1和奖池2，均对空艇格提升500权重
  奖池1空艇类剩余50分值，奖池2空艇类剩余200分值
  
  → 空艇格权重总共提升1000
  → 触发空艇格时，奖金浮动倍率取200分值对应的倍率区间
  → 扣除时先扣奖池1的50分值，再从奖池2扣除剩余
  → 之后空艇格权重仅提升奖池2的500
```

### 4.4 概率提升规则

#### 4.4.1 核心公式

```
P_boost = P_base × (1 + K × V_pool / (E_val × Bet))

其中：
  P_base   = 事件基础触发概率
  K        = 调节系数（配置值）
  V_pool   = 奖池剩余分值
  E_val    = 事件平均期望分值（含动态价值系数）
  Bet      = 玩家当前下注价值
```

#### 4.4.2 概率提升计算流程

```
// Step 1: 计算奖池水位比例
waterLevelRatio = pool.remainingValue / pool.initialValue

// Step 2: 计算基础提升倍率
baseBoostMultiplier = 1 + pool.K × waterLevelRatio

// Step 3: 应用上限保护
adjustedProbability = P_base × baseBoostMultiplier
adjustedProbability = min(adjustedProbability, P_base × 3.0)
adjustedProbability = min(adjustedProbability, 0.5)
```

#### 4.4.3 概率提升配置示例

| 奖池水位 | K=1.0时 | 铁路格P_base=0.08 → | 说明 |
|---------|---------|-------------------|------|
| 100%（满池） | 1+1.0=2.0 | 0.16 | 概率翻倍 |
| 60% | 1+0.6=1.6 | 0.128 | 温和提升 |
| 30% | 1+0.3=1.3 | 0.104 | 小幅提升 |
| 接近0 | 1+0=1.0 | 0.08 | 回归基础 |

### 4.5 价值扣除规则（奖池消亡机制）

#### 4.5.1 核心逻辑

每次玩家触发奖池关联事件并获得奖励时，从奖池中扣除对应分值。**判断依据是事件实际产出价值（EventValue），不是玩家Bet。**

```
EventValue = 事件实际产出价值（金币+代币×Rate_token+骰子×Rate_dice）
Min = pool.minValue      // 配置值：事件产出价值下限
Max = pool.maxValue      // 配置值：事件产出价值上限
alpha = pool.alpha       // 低产出加速系数，建议0.3~0.5
beta = pool.beta         // 高产出缓释系数，建议0.3~0.5

if EventValue < Min:
    // 事件产出太低，加速消耗奖池，避免低效磨池
    actualDeduction = EventValue × (Min / EventValue) ^ alpha
    // 例：EventValue=2, Min=10, alpha=0.5 → 扣除 = 2 × (10/2)^0.5 = 3.16
    // 奖池扣得比事件产出快，加速消亡

elif EventValue > Max:
    // 事件产出太高，缓扣延长奖池寿命
    actualDeduction = Max + (EventValue - Max) × beta
    // 例：EventValue=150, Max=50, beta=0.5 → 扣除 = 50 + 100×0.5 = 100
    // 玩家拿了150价值，奖池只扣100，产生延续性

else:
    // 正常区间：按实际价值扣除
    actualDeduction = EventValue

// 更新奖池剩余
pool.remainingValue -= actualDeduction
```

#### 4.5.2 参数建议值

| 参数 | 建议值 | 说明 |
|------|--------|------|
| Min | 事件期望价值的50%~80% | 低于此值视为低效产出 |
| Max | 事件期望价值的150%~200% | 高于此值视为高产出 |
| alpha | 0.3~0.5 | 越小加速越温和 |
| beta | 0.3~0.5 | 越小延续性越强 |

#### 4.5.3 特殊配置

某些奖池可以不设置Max上限（配置Max=0或-1），允许一次性掏空。适用于：
- 充值大额返利奖池
- 破产恢复救济池
- 活动最终结算池

#### 4.5.4 坏账预留机制

```
生成奖池预算时，预设 5% 的溢出空间：

initialValue = 计划发放价值 × 1.05
overflowReserve = initialValue × 0.05

目的：
  - 对冲Max机制带来的超发（玩家实际获得 > 奖池扣除）
  - 确保即使出现超额发放，也不会影响基础RTP底线
```

### 4.6 奖池死亡条件

满足以下任一条件，奖池死亡：

```
条件1：奖池被掏空（含溢出消耗）
    pool.remainingValue <= 0

条件2：掷骰次数达到上限
    currentDiceCount - pool.initialDiceCount >= pool.maxDiceCount

条件3：被更高优先级池顶替
    // 可选：如果设计了优先级抢占机制
```

死亡后处理：
```
1. 标记pool.isActive = false
2. 从优先级队列中移除
3. 激活下一个优先级的池（如有）
4. 如果队列为空，所有概率修正归零，回归基础权重
```

### 4.7 抽水池规则

#### 4.7.1 定位调整

抽水池(-)不再作为"还债机制"（即不因玩家赢多了而触发），改为：

**新手期后的难度调节工具**

| 属性 | 说明 |
|------|------|
| 触发条件 | 玩家背包里的骰子存量远超当前等级进度时 |
| 设计目的 | 防止玩家攒太多资源导致后续付费点失效 |
| 不是 | 惩罚玩家"运气好" |

#### 4.7.2 抽水池入队逻辑

```
// 资源存量监控
if player.diceInventory > currentLevel.recommendedDice × 1.5:
    // 骰子存量超过推荐值的150%
    小概率触发抽水池入队
    入队概率 = f(超出比例, 玩家付费历史)
    
    // 超出越多、付费意愿越低的玩家，入队概率越高
```

#### 4.7.3 抽水池效果

```
概率调整：
  对应事件触发概率降低
  降低幅度随剩余分值衰减（与奖池对称）

价值回收：
  通过降低事件成功概率或降低奖励倍率实现
  不直接从玩家已有资产中扣除

死亡条件：
  同奖池（回收到上限 或 掷骰次数达到上限）
```

### 4.8 Min/Max造成的差额处理

```
Min/Max机制必然导致玩家实际获得与奖池扣除存在差额：
  - EventValue < Min时：奖池扣除 > 玩家获得（加速消耗）
  - EventValue > Max时：奖池扣除 < 玩家获得（延续性来源）

处理策略：不单独"还债"，依赖以下机制自然回收：
  1. 基础RTP的底线守卫（无奖池时自然损耗）
  2. 周期性RTP回调机制（偏差超阈值时调整下期参数）
  3. 奖池预算的5%溢出预留（对冲Max超发）
```

## 5. 保底机制

### 5.1 轨道格保底

```
机制：玩家连续N次未踩中某类格子时，强制下一次命中

实现：
  为每类格子维护一个计数器 counter[eventType]
  每次掷骰后：
    如果落点是该类型格子 → counter[eventType] = 0
    否则 → counter[eventType]++
  
  当 counter[eventType] >= threshold[eventType] 时：
    下次掷骰Step 1选格时，将该类型格子权重临时提升10倍
    （或直接指定落点为该类型格子中最近的一个）
```

| 格子类型 | 保底阈值(建议) | 说明 |
|---------|--------------|------|
| 社交格(空艇) | 15-20骰 | 保证社交体验频率 |
| 机会格(?) | 10-15骰 | 保证随机事件体验 |
| 端点格 | 20-25骰 | 保证小游戏触发 |
| 租户格 | 8-12骰 | 保证收租体验 |

### 5.2 终点保底

```
到达地图终点时：
  100%触发奖励事件
  给予阶段完成感（额外骰子/大额money/稀有道具等）
```

### 5.3 机会格直达空艇保底

```
机会格(?)抽取事件中，"直达番茄空艇"事件纳入保底
连续X次未抽中直达空艇时，下次抽取权重提升
```

### 5.4 保底与RTP的协调

```
保底触发时：
  1. 不临时调整当前奖池参数
  2. 保底强制命中的格子，其奖励按正常规则结算
  3. 如果该格子类型在当前奖池的加分事件中，正常消耗奖池分值
  4. 长期来看，保底造成的RTP偏差会被周期性回调机制修正
```

## 6. 事件系统

### 6.1 机会格(?)事件

```
触发：踩中机会格(8, 23, 34, 48, 58)
效果：抽取一张机会牌，触发随机事件

事件类型（按权重抽取）：
  - 赚钱（钱1/钱2/钱3，不同倍率）
  - 赚骰子
  - 直达番茄空艇（纳入保底）
  - 其他特殊事件

抽取规则：
  根据配置权重随机
  同一类型内区分多种倍率和包装结果
  受当前奖池/抽水池影响（概率提升/降低）
```

### 6.2 端点格小游戏

| 端点格 | 小游戏 | 规则 |
|--------|--------|------|

### 6.3 特殊关联格

```
编号：5, 13, 27, 40, 44, 59
常规效果：获取或扣除一笔钱
特殊规则：
  - 获取的钱数 = 基础值 × bet倍率
  - 扣除的钱数：最大不超过玩家当前拥有的X%，最低不低于Y值
  - 与地图等级相关
  - 活动中用于发放固定位置的活动代币
```

## 7. 反作弊机制

### 7.1 服务端校验

```
所有骰子结果必须在服务端生成
客户端仅负责表现层动画
客户端上报的操作必须经过服务端验证：
  - 掷骰请求 → 服务端返回结果（落点格、骰子值、奖励）
  - 客户端动画根据服务端结果播放
```

### 7.2 异常检测

```
监控指标：
  1. 玩家长期RTP显著偏离基准值（>±30%）
  2. 特定格子触发频率异常高
  3. 短时间内获得异常大量奖励
  
处理策略：
  1. 记录异常日志
  2. 触发人工审核
  3. 必要时临时限制操作频率
```

### 7.3 随机数安全

```
使用加密安全的随机数生成器（如CSPRNG）
随机数种子定期轮换
关键随机过程（骰子结果、事件抽取）记录种子和结果用于审计
```

## 附录A：算法流程图

```
玩家点击掷骰
    │
    ▼
服务端校验骰子数量是否足够
    │
    ├── 不足 → 返回错误，提示购买或等待
    │
    ▼ 足够
扣除骰子数量
    │
    ▼
检查当前生效奖池/抽水池
    │
    ▼
计算P_boost概率提升
    │   P_boost = P_base × (1 + K × V_pool / (E_val × Bet))
    │
    ▼
检查保底步数计数器
    │   if consecutiveHits >= 3 → 强制落点为平庸格
    │
    ▼
Step 1: 计算前方2-12格候选权重
    │   (基础权重 × 奖池修正 × 保底修正)
    │
    ▼
随机选择目标格
    │
    ▼
Step 2: 反推骰子点数和
    │   (distance = target - current)
    │   if distance ∉ [2,12] → 重试Step 1
    │
    ▼
Step 3: 随机化骰子具体值
    │
    ▼
Step 4: 结算格子事件
    │   ├── 连接格 → 传送 + money
    │   ├── 端点格 → 小游戏
    │   ├── 社交格 → 抢劫/宝箱
    │   ├── 租户格 → 收租
    │   ├── 机会格 → 抽卡
    │   ├── 特殊关联格 → 地图关联奖惩
    │   └── 普通格 → money
    │
    ▼
计算EventValue（含动态价值系数）
    │   EventValue = Gold×1 + Token×Rate_token + Dice×Rate_dice
    │
    ▼
更新奖池分值
    │   if EventValue < Min → 扣除 = EventValue × (Min/EventValue)^α
    │   if EventValue > Max → 扣除 = Max + (EventValue-Max)×β
    │   else → 扣除 = EventValue
    │
    ▼
更新保底步数计数器
    │   if 奖池关联格 → consecutiveHits++
    │   else → consecutiveHits = 0
    │
    ▼
检查奖池是否死亡
    │   ├── 是 → 激活下一个优先级池
    │   └── 否 → 保持当前池
    │
    ▼
记录RTP数据
    │
    ▼
返回结果给客户端
    │
    ▼
客户端播放动画
```
